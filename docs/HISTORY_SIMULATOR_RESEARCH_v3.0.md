## 🏗️ 后端架构 (深度拆解)

### 1. 三段式推演管线 (来自新浪财经)

```
玩家写诏书 → 千问/GPT 推演:
  ① 意图解析     (自然语言 → 结构化指令)
  ② 调用数据     (国库/民心/军队/派系/地方等上千维度)
  ③ 逻辑推演     (因果链 + 历史规律)
  ④ 数据更新     (回写数据库 + 玩家反馈)
```

### 2. 数据库角色 (新浪财经原文)

> **"以 PolarDB 为代表的数据库不再只是'仓库'，而是支撑 AI 思考和决策的数据中枢。它需要将自然语言实时转换为结构化数据，从而记住每个玩家的游戏世界状态，追踪每位大臣不断变化的长期记忆；不仅如此，数据库还要在极短时间内完成向量检索等任务，把正确的数据喂给大模型，驱动下一步推演。通过数据压缩等技术优化，存储成本降低了 50% 以上。"**

### 3. 多 Agent 框架 (Steam 官方 Q&A)

> **"由于本游戏的 Agent 框架相对复杂，例如推演流程有多个 Agent 运行，以保证游戏尽可能维持准确性，此外，请确保您的 API 支持 KV cache 以避免过高的输入成本。"**

→ **关键启示**：推演流程是**多 Agent 协作**（不是单 Agent 一把梭），需要 KV cache 优化。

### 4. API Key 本地化 (Steam 官方 Q&A)

> **"玩家的 API Key 不会被上传到服务器，这也是我们做的主要改动之一，基于 API 的调用将从用户本地发起，API Key 将全程保持在用户本地。"**

→ **关键启示**：v3.0 阶段**必须实现本地 API Key 路由**，避免服务端代收导致合规问题。

### 5. 已接入模型 (官方)

## 🧠 提示词工程 (System Prompt 模板)

### 1. ZEGO 即构科技 角色扮演模板 (来自官方文档)

```markdown
# Role: [角色名]
# Background: [历史背景 / 时代背景 / 个人经历]
# Personality: [性格特征, 用具体例子佐证]
# Speaking Style: [语言风格, 文言/白话/口头禅]
# Relationships: [与玩家关系, 与其他 NPC 关系]

## Goals
- 第一目标: [...]
- 第二目标: [...]

## Constraints
- 必须遵守: [...]
- 不得违反: [...]

## Workflow
1. 接收皇帝诏书
2. 解析诏书意图
3. 调用相关数据
4. 输出: 奏折 (申论式分析) + 数值变化建议
```

### 2. 火山方舟 设计原则 (实战经验)

> **"人设信息维度 = 性格 + 背景 + 经历 + 时代 + 说话方式"**
> **"通过提示词优化, 模型回复从生硬逐渐自然贴近日常"**

### 3. 智谱 推荐配置 (GLM-5 模式)

```yaml
角色深度:
  性格: 3-5 个具体特征 (例: 谨慎/刚直/权变/猜忌)
  立场: 派系 + 阶级 双维度
  知识: 历史常识 + 专业知识
  弱点: 把柄 / 软肋 (用于反弹/曲解/拖延)

输出格式:
  奏折: 文言文 4-6 段 (申论式)
  数值: JSON 结构 (国库 -X / 民心 +Y / 派系关系 Z)
  建议: 3 条可选诏书方向

历史一致性:
  时间线: 当前年份 + 历史事件
  人物: 已发生 + 未发生 (不出戏)
  地理: 行政区划 + 距离
```

### 4. 长上下文防"模型幻觉" (青干 Steam 更新日志)

> **"优化了「准奏」功能的提示词与代码模块，令输出内容更聚焦于当前回合的议题，减少了因长上下文引发的模型幻觉问题。"**

→ **关键启示**：必须**每个回合注入"当前议题"**作为提示词硬约束，否则长上下文中模型会跑题/编造。

### 1. 整体结构 (来自评测原文)

```
┌──────────────────────────────────────────────────────┐
│ 顶部: 国势仪表盘 (国库/民心/军队/地方/派系)         │
├──────────────────────────────────────────────────────┤
│ 主区: 时政任务卡 (当前危机描述 + 6 困境图)          │

├──────────────────────────────────────────────────────┤
│ 诏书输入: 多行文本框 + AI 建议按钮                   │
├──────────────────────────────────────────────────────┤
│ 反馈: 申论式分析 + 数值变化 + 新回合提示             │
└──────────────────────────────────────────────────────┘
```

### 2. 关键 UI 元素 (评测原文摘录)

| 元素 | 描述 |
|------|------|
| **开局六大困境** | 卡片网格, 每张卡描述一个危机 (国库赤字/陕西大旱/各地战乱/边患/阉党/流民) |
| **大臣奏折** | 文言文 + 头像 + 派系标识, 申论式分析当前议题 |
| **AI 建议诏书** | 旁路按钮, "采纳 AI 建议"一键填入诏书框 |
| **威望驱动** | 数字大屏: 威望 > 80 政令如雷霆 / 威望 < 30 阳奉阴违 |
| **阶层满意度** | 多维度条形图 (农民/士绅/商人/军户) |
| **科技树** | 树状图 + 节点 (农业/财政/军事/工业), 后期可"穿越"工业革命 |
| **存档系统** | 主动存档 + 实时快照 (修复了 5-19 之前的存档 Bug) |

### 3. UI 痛点 (差评集中)

| 痛点 | 描述 |
|------|------|
| **美工 AI 化** | 玩家一眼看出"AI 画的", 美术缺乏精修 |
| **Token 计费不透明** | 玩家吐槽"积分是负数, 倒欠功德值" |
| **存档丢失** | 早期版本存档问题导致进度丢失 (5-19 修复) |
| **模型幻觉** | 同一 NPC 不同回合时间线对不上 (曹化淳事件) |
| **长上下文成本** | 一次推演 0.1 千 token ≈ 1 元, 烧钱 |

## 💰 Token 经济学 (核心商业痛点)

### 1. 定价 (青干官方)

| 项 | 价格 |
|----|------|
| Steam 本体 | **48 元** |
| 推演额度包 | 按 token 计费, 1 回合 ≈ 0.1 千 token ≈ 1 元 |
| API 自接 | 用户自带 Key, 本地调用, 不上传 |

### 2. 成本困境 (制作人原话, 36氪)

> **"AI 大模型在游戏应用中体验与成本的'跷跷板困境'：两者永远只能落地一个。"**

### 3. 三大降本手段 (实测有效)

| 手段 | 来源 | 效果 |
|------|------|------|
| **KV cache** | Steam Q&A 官方建议 | 降低输入 token 重复计费 |
| **数据压缩** | 新浪财经 (PolarDB) | 存储成本 **降 50%** |
| **多模型适配** | 官方 3 家 | 用户可自选低成本模型 (DeepSeek 0.11 美元/百万 token) |
