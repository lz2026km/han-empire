## 2. 竞品核心发现 (5 大优势 + 1 致命问题)

### 2.1 优势 #1: score_extractor 5 档房拆分 (竞品独有)
**竞品 v0.4.6**:
- `score_extractor.md` (311 行) - 主控
- `score_extractor_shared.md` (64 行) - 字段共享规则
- `score_extractor_internal.md` (67 行) - 内政财政
- `score_extractor_issues.md` (78 行) - 局势推进
- `score_extractor_military_external.md` (90 行) - 军外
- `score_extractor_personnel_secret.md` (71 行) - 人事密令

**关键设计模式**: 每个档房只允许输出**自己字段** (铁律字段所有权)
- internal 只输出: `国势变化` / `钱粮收支` / `财政制度变化` / `派系变化` / `阶级变化` / `地区变化`
- issues 只输出: `局势推进` / `新立局势` / `撤销局势` / `结案局势`
- military_external 只输出: 军/外族
- personnel_secret 只输出: 人事/密令

**我方 v4.9 现状**:
- `score_extractor.md` (311 行, **单体**, 全部字段都允许)
- 字段所有权**不明确** → 容易越界

**v5.0 改造方案**: P0-1 拆 5 档房

### 2.2 优势 #2: 推演/聊天/简报 3 模型分级 (竞品独有)
**竞品 v0.4.6** (5/28 Steam API 测试版):
- **Simulate_model (推演)**: Qwen-3.6-Plus / Deepseek V4 Flash (推荐), GLM-5 验证
- **Chat_model (聊天)**: Qwen-3.5-plus / Qwen Plus / Deepseek V4 Flash (中等尺寸)
- **Briefing_model (推测)**: 简报类用中等尺寸

**我方 v4.9 现状**:
- `model_adapter.py` 支持 4 个 provider (Qwen/MiniMax/DeepSeek/OpenAI)
- `api_key_router.py` 支持 1 个 key 模式 + server fallback
- **没有按"用途"分级** → 推演/聊天/简报用同一模型

**v5.0 改造方案**: P0-2 实现模型分级路由

### 2.3 优势 #3: 默会知识 + 工具调用铁律 (竞品 prompts 风格)
**竞品 prompts 独有模式**:
- "靠的是默会知识 (Michael Polanyi tacit knowledge)" - 强调笔法靠模型自己悟
- 工具调用铁律: "提及某人是否在职 → 必须先调 `inspect_minister(name)` 取实值, 不得凭记忆"
- 字段所有权铁律: "本模块只允许输出这 6 个字段, 严禁输出军队/势力/..."

**我方 v4.9 prompts 风格**:
- ✅ 也有"权威源声明" (game_world 为准)
- ❌ 缺"默会知识"指导语
- ❌ 缺工具调用铁律

**v5.0 改造方案**: P1-1 prompts 升级 (仿竞品补这两类指令)

### 2.4 优势 #4: 详细的输入契约 (竞品 prompts 显式)
**竞品 season_simulator.md 头部**:
- 显式列出 input 字段: `regions` / `armies` / `buildings` / `active_issues` / `previous_narrative_tail` / `historical_anchor` / `victory_status` / `fixed_flows` / `candidate_events` / `secret_orders`
- 每个字段给类型说明

**我方 v4.9 simulator.md 头部**:
- 已有盘面上下文 Header 格式 (✅ 已有)
- 缺: 各字段类型 / 强约束

**v5.0 改造方案**: P1-1 (合并到 P0-1)

### 2.5 优势 #5: 反推演链条 (竞品独有)
**竞品 event_selector.md 模式**:
- 接收 `candidates` 列表, 每条含 `id`/`title`/`kind`/`summary`/`interests`/`is_historical`/`resolve_condition`/`fail_condition`/`precondition`
- precondition 字段**显式说明**玩家可如何改写/免除此事

**我方 v4.9 现状**:
- `event_selector.md` 146 行, 已有基础
- 缺: `precondition` 字段的玩家改写口子

**v5.0 改造方案**: P1-2

### 2.6 致命问题: 54% 好评率 (主公可避坑)
- 36氪报道: 烧 Token 成本太高
- Steam 差评集中: "6 年 token 见底" / "服务器卡" / "自由度不足"
- 玩家 70 小时造出原子弹 = **完全偏离历史**

**v5.0 应对**:
- 推演用 flash 模型降本
- 强提示词限制"穿越式"脑洞
- token 实时监控, 提前预警

- `han_sim/score_extractor_pipeline.py` (~150 行) - 5 档房串联调用器
  - 接口签名:
    ```python
    def extract_score(narrative: str, ctx: dict) -> dict:
        """5 档房串行抽取 + 合并 JSON
        1. shared 抽取 (字段共享规则)
        2. internal 抽取 (内政财政)

from han_sim.score_extractor_pipeline import extract_score

narrative = season_sim_result["narrative"]
score = extract_score(narrative, ctx={"regions": ..., "armies": ...})
# score 是个 dict, 含 metric_delta / faction_delta / region_delta / ...
```

**验收标准**:

- [ ] score_extractor_pipeline.py 单测 ≥ 20 个 case 通过

- [ ] 失败回退: 故意让 1 个档房 prompt 报错 → 系统回退到单体

**预期收益**: 数值抽取错误率 -50% (字段所有权清晰), 单次 token 用量 -30% (5 个短 prompt vs 1 个长 prompt)

#### P0-2: 模型分级路由 (推演/聊天/简报 3 模型)

**目标**: 按用途分配不同 model, 推演用 flash 降本, 角色用 plus 提质

**新增文件** (1 个):
- `han_sim/llm_router.py` (~150 行) - 模型路由器
  - 接口签名:
    ```python
    class ModelTier(str, Enum):
        SIMULATE = "simulate"  # 推演 (量最大, 用 flash)
        ROLEPLAY = "roleplay"  # 角色 (大臣/妃嫔, 用 plus)
        BRIEFING = "briefing"  # 简报 (用 flash)
        SANITIZE = "sanitize"  # JSON 清洗 (用 flash)

    class LLMRouter:
        def __init__(self, config: dict):
            """config 例:
            {
              "simulate": "qwen-flash",
              "roleplay": "qwen-plus",
              "briefing": "qwen-flash",
              "sanitize": "qwen-flash"
            }
            """

        def get_model(self, tier: ModelTier) -> str:
            """返回该 tier 用的 model 名字"""

        def chat(self, tier: ModelTier, req: ChatRequest) -> ChatResponse:
            """按 tier 路由到对应 model, 自动加 cache"""
    ```

**修改文件** (9 个 agent 工厂):
- `han_sim/agents.py` 中 9 个 agent 构造函数: 加 `model_tier: ModelTier = ModelTier.ROLEPLAY` 参数
  - `create_minister_agent` → ROLEPLAY (大臣)
  - `create_season_simulator_agent` → SIMULATE (推演)
  - `create_score_extractor_agent` → SIMULATE (抽取)
  - `create_consort_agent` → ROLEPLAY (妃嫔)
  - `create_event_selector_agent` → BRIEFING
  - `create_chat_memory_agent` → BRIEFING
  - `create_json_sanitizer_agent` → SANITIZE
  - `create_memory_retrieval_agent` → SANITIZE
  - (新) `create_score_pipeline_agent` (P0-1 配套) → SIMULATE

**配置扩展** (`runtime_llm.json` 新增):
```json
{
  "model_tiers": {
    "simulate": "qwen-flash",
    "roleplay": "qwen-plus",
    "briefing": "qwen-flash",
    "sanitize": "qwen-flash"
  }
}
```

**验收标准**:
- [ ] llm_router.py 单测覆盖 4 tier × 3 model = 12 case
- [ ] 9 个 agent 工厂全部接 model_tier 参数
- [ ] E2E: 跑 5 回合, 验证:
  - 推演用 flash (低成本)
  - 角色用 plus (高质量)
  - 总 token 成本比 v4.9 降 ≥ 30%
- [ ] 向后兼容: 不配 model_tiers → 全部用 ROLEPLAY (默认)

**工作量**: 2 人天 (1 个 router 0.7 + 9 个 agent 改造 0.7 + E2E 0.6)
**预期收益**: token 成本 -30~50% (推演量最大, 用 flash 降 60% 单价)

## 7. 风险与缓解 (具体)

| 风险 | 等级 | 触发条件 | 缓解方案 |

| 玩家抱怨 token 反而多 | 高 | 仪表盘显示高价 | 加单价开关, 默认用最低价模型 |
